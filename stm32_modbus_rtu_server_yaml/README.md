# STM32 Modbus Server (modbus-rs Demo)

## Overview
This project is a demonstration for developers who want to integrate the [modbus-rs](https://github.com/Raghava-Ch/modbus-rs) project into a bare-metal C environment. It implements a Modbus RTU Server (Slave) on an STM32 microcontroller. The server leverages your robust, safe Rust-based `modbus-rs` stack (integrated via C-bindings generated by `cbindgen`) to handle Modbus frames and transactions natively in C.

The application is designed around a non-blocking super-loop architecture and features interrupt-driven UART communication, a ring buffer for receiving data, and non-blocking transmissions. It provides practical examples of hardware integration by mapping Modbus coils and registers to the on-board LEDs (Green, Red, Yellow).

## Modbus Memory Map & Hardware Integration
This demo maps incoming Modbus frames directly to hardware actions and system variables based on the definitions in the [server.yaml](./CM7/modbus_server/server.yaml) specification. Below are the elaborated addresses, access permissions, and their hardware interactions:

### 🟡🟢🔴 Coils (Read/Write & Write-Only)
Used to directly turn the onboard Nucleo LEDs on or off. Note the specific access controls applied by the stack.

| Address | Name           | Access | Description                         | Hook Function             |
|---------|----------------|--------|-------------------------------------|---------------------------|
| `0x0000`| **Yellow LED** | **RW** | Write `1` to turn ON, `0` for OFF   | `app_on_write_led_yellow` |
| `0x0001`| **Red LED**    | **RW** | Write `1` to turn ON, `0` for OFF   | `app_on_write_led_red`    |
| `0x0002`| **Green LED**  | **RW** | Write `1` to turn ON, `0` for OFF   | `app_on_write_led_green`  |

### 🔘 Discrete Inputs (Read-Only)
Used to read the state of onboard hardware inputs, such as the Nucleo user button.

| Address | Name           | Access | Description                                 |
|---------|----------------|--------|---------------------------------------------|
| `0x0000`| **User Button**| **RO** | Read `1` when PRESSED, `0` when RELEASED    |

Tip: Press and hold the user button to see the discrete input state change in your Modbus client.

### ⏱️ Holding Registers (Read/Write)
Used to control numerical values and timing variables in the super-loop.

| Address | Name           | Access | Description                                  | Hook Function              |
|---------|----------------|--------|----------------------------------------------|----------------------------|
| `0x0000`| **Blink Delay**| **RW** | Sets the LED toggle interval in milliseconds | `app_on_write_blink_delay` |

Value `0` means no blinking (LEDs will be static based on coil states). Non-zero values will cause the LEDs to toggle at the specified interval.

## Features
* **Modbus RTU Protocol**: Serial Modbus server responding to unit ID `1`.
* **Rust-powered Stack (`modbus-rs`)**: Core Modbus logic is handled by a safe Rust library, interfaced via `modbus_rs_server.h`.
* **Non-Blocking Serial I/O**:
  * **RX**: Interrupt-driven byte-by-byte reception (`HAL_UART_RxCpltCallback`) feeding into a volatile lock-free ring buffer.
  * **TX**: Non-blocking transmission using `HAL_UART_Transmit_IT`, with application-level busy state checking.
  * **Error Recovery**: Automatic recovery from UART Overrun Errors (ORE) via `HAL_UART_ErrorCallback`.
* **Hardware Control**: Read/Write Modbus coils and registers trigger callbacks to control STM32 Nucleo LEDs.

## Hardware Configuration
* **Microcontroller**: STM32H7 Series
* **UART Peripheral**: `USART3`
* **Baud Rate**: 115200 bps
* **Data Bits**: 8
* **Parity**: None
* **Stop Bits**: 1
* **Pins**: 
  * `PD8` -> USART3_TX
  * `PD9` -> USART3_RX

## Concurrency & Thread Safety
Currently, the application runs entirely in a bare-metal, single-threaded context. The Modbus server execution is driven by polling `mbus_serial_server_poll()` in the main `while(1)` loop. 

Stub locks (`mbus_app_lock`, `mbus_pool_lock`, etc.) are provided in `modbus.c`. If an RTOS (e.g., FreeRTOS) is integrated in the future, these stubs should be populated with Mutexes or `__disable_irq()` / `__enable_irq()` to prevent race conditions during state mutations.
